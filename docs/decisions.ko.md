# Decisions

libre의 방향을 결정하는 선택들을 간단히 기록합니다. 새로운 의존성과 아키텍처 관련
결정은 여기에 기록합니다(CLAUDE.md 참고).

## Protocol: Nostr, NIP-72

Nostr는 클라이언트가 소유하는 키와 단순하고 교체 가능한 릴레이를 제공하며, 이는
"중앙 게시물 삭제 지점이 없다"는 목표에 가장 잘 들어맞습니다. NIP-72는 레딧 스타일의
모더레이션 커뮤니티를 모델링합니다(`kind 34550` 커뮤니티, 게시물/답글용 `kind 1111`
NIP-22 코멘트, `kind 4550` 승인). 검토했던 대안: ActivityPub/Lemmy(연합형이지만 운영자
병목 지점 존재), Farcaster(하이브리드, 온체인 의존성 존재). 단순성과 검열 저항성 면에서
Nostr가 선택되었습니다.

## Frontend: vanilla JS + HTML, no framework

요구 사항: 빠르게 만들 수 있고, 단순하며, Hacker News처럼 꾸밈없고, "AI 슬롭" 같은
디자인이 아닐 것. 프레임워크(React/Vue)는 여기서는 이점 없이 빌드 복잡성과 더 넓은
공격 표면만 추가합니다. 앱 규모가 작으므로 순수 ES 모듈과 작은 DOM 헬퍼만으로 충분하며,
정적 파일이라는 불변 조건도 자연스럽게 유지됩니다.

## Dependencies

- **nostr-tools** (MIT): 유일한 런타임 라이브러리입니다. 검증되고 감사받은 암호화/릴레이
  코드이므로 서명, bech32, 릴레이 프레이밍을 직접 구현하지 않습니다. **벤더링됨**: esbuild로
  한 번 번들링되어 `web/vendor/nostr-tools.js`(커밋됨)에 포함되므로 런타임에는 npm도
  CDN도 필요하지 않습니다.
- **esbuild** (dev only): 벤더 파일을 번들링합니다. 배포에는 포함되지 않습니다.
- **JetBrains Mono** (SIL OFL 1.1): UI 폰트. **벤더링됨**: `.woff2` 파일이
  `web/vendor/fonts/`에 커밋되어 있으며, 개발 환경에서는 `@font-face`와 상대 `url()`로
  로드하고, 단일 파일 배포본에서는 `build.js`가 base64 `data:` URI로 인라인 처리합니다.
  런타임에 Google Fonts나 CDN 요청이 없습니다.
- 그 외 런타임 의존성은 없습니다. 새로 추가하려면 여기에 기록해야 하며, 런타임 서드파티
  호스트를 도입해서는 안 됩니다.

## Relay: strfry

검증된 C++ 릴레이로, Docker 배포가 쉽고 Blossom 미디어 서버 역할도 겸합니다. NIP-72는
릴레이 측의 특별한 지원을 필요로 하지 않으므로(릴레이는 그저 해당 이벤트 kind를
저장할 뿐입니다) 표준을 준수하는 어떤 릴레이든 사용할 수 있으며, strfry가 권장 기본값입니다.
자동 WSS를 위해 Caddy를 앞단에 둡니다.

## Licensing: AGPL-3.0-or-later

네트워크 카피레프트: 수정된 libre를 서비스로 운영하는 사람은 누구나 변경 사항을
공개해야 하므로 포크가 계속 열려 있게 되고 네트워크가 함께 성장합니다. libre는 서드파티
앱 코드를 포함하지 않습니다. `reference/chorus`는 학습용 참고 자료일 뿐이며 배포에서
제외됩니다(NOTICE, .gitignore 참고).

## i18n: JSON tables + `t()`, English source of truth

전 세계 사용자를 대상으로 합니다. `web/i18n/<locale>.json`에 평탄한 점 표기 키를 사용하고,
날짜에는 `Intl`을, RTL에는 논리적 CSS 속성을 사용합니다. 언어를 추가하는 것은 JSON 파일
하나를 추가하는 것과 같습니다. `en`과 `ko`를 함께 출시합니다.

## Deferred (not in MVP)

- 앱 내 모더레이션 UI(승인/차단/신고): 프로토콜 상의 로직은 존재하지만
  (`kind 4550/4551/34553`) 클라이언트 UI는 향후 작업입니다.
- 빠르게 전환 가능한 다중/일회용 신원.

## Shipped after MVP

- **네임스페이스 필터**: 커뮤니티는 `["t","libre"]` 태그를 가지며, 목록은
  `#t:["libre"]`로 조회하므로 libre는 공유 릴레이의 모든 kind-34550이 아니라 자신이
  만든 커뮤니티만 보여줍니다. 이는 발견(discovery)용 필터이지 보안 경계가 아닙니다.
- **게시물 삭제**: 자신의 게시물에 대한 NIP-09(`kind 5`) 요청입니다. 최선을 다하는
  방식일 뿐입니다. 이를 준수하는 릴레이는 이벤트를 삭제하지만, 다른 곳에 남은 사본은
  살아남을 수 있습니다.
- **이미지 업로드**: Blossom(BUD-02) 방식으로 sha256 콘텐츠 주소 기반 블롭을 사용자가
  직접 편집 가능한 미디어 서버 목록에 업로드합니다(기본값 blossom.band, blossom.nostr.build).
  게시물 안의 이미지 URL은 `<img>`로 렌더링됩니다. 인증은 서명된 kind-24242 이벤트로
  이루어집니다. 업로드 전에 이미지는 canvas를 거쳐 다시 인코딩되어 EXIF(GPS/기기 정보)를
  제거합니다.
- **로컬 키 암호화**: 로컬 nsec는 더 이상 평문으로 저장되지 않습니다. PBKDF2(210k)
  기반 패스프레이즈 키로 AES-GCM 암호화되며, 암호문은 IndexedDB에 저장됩니다. 패스프레이즈
  자체는 저장되지 않고 잠금 해제를 위해 매번 다시 입력해야 합니다(세션 동안만 메모리에
  유지됨). 기존의 평문 localStorage 키는 로드 시 삭제됩니다. NIP-07이 여전히 가장 안전한
  방법입니다. 로그인 시 이제 IP 노출(Tor/VPN 사용 권장)과 게시물의 영구성에 대해 경고를
  표시합니다.
- **답글의 스레드 루트 `E` 태그**: chorus는 모든 코멘트의 루트를 커뮤니티(`A` 태그)로만
  잡기 때문에, 부모 체인 전체를 수신하지 못하면 답글을 원래 게시물에 연결할 수 없었습니다.
  두 사용자의 릴레이 집합이 완전히 겹칠 필요는 없으므로, 부모 하나가 누락되면 하위 트리
  전체가 조용히 사라지는 문제가 있었습니다. 이제 libre의 답글은 `["E", <최상위 게시물 id>]`
  (NIP-22 루트 이벤트)도 함께 가집니다. 코멘트 뷰는 이런 고아 답글을 "미수신 부모" 안내와
  함께 최상위 레벨에 고정합니다. 이 변경 이전(또는 chorus에서 생성된) 이벤트에는 이 태그가
  없어 계속 연결할 수 없는 상태로 남습니다. 릴레이 설정 페이지는 릴레이가 하나만 설정된
  경우에도 경고를 표시합니다. 릴레이가 하나뿐이면 단일 장애점이자 검열과 감시에 취약한
  지점이 되기 때문입니다.
